## Q2. At which OSI layer does Azure Load Balancer operate?

**Answer (explained):**
Azure Load Balancer operates at **Layer 4 (Transport Layer)** of the OSI model. This means it makes routing decisions based on IP addresses and ports using protocols like TCP and UDP, rather than inspecting application-level data.

---

## Q3. How does Azure Load Balancer decide where to send traffic?

**Answer (explained):**
Azure Load Balancer uses **load-balancing rules** combined with **health probes**. It continuously checks backend instances for health and only routes traffic to those that are responsive and healthy. If a backend VM fails health checks, traffic is automatically redirected to healthy instances.

---

## Q4. What are health probes in Azure Load Balancer?

**Answer (explained):**
Health probes are monitoring mechanisms that regularly check the availability and responsiveness of backend VMs. They send probe requests at defined intervals to verify whether a VM is healthy. If a VM fails to respond correctly, it is temporarily removed from traffic rotation.

---

## Q5. What protocols are supported by Azure Load Balancer health probes?

**Answer (explained):**
Health probe protocol support depends on the Load Balancer SKU:

* **Standard SKU** supports **TCP, HTTP, and HTTPS**
* **Basic (Essential) SKU** supports **TCP and HTTP**

HTTPS probes require certificates with **SHA256** signature across the entire certificate chain.

---

## Q6. How do TCP and HTTP/HTTPS probes differ?

**Answer (explained):**
A **TCP probe** attempts to establish a TCP handshake. If the handshake succeeds, the VM is considered healthy.
An **HTTP/HTTPS probe** sends an HTTP GET request to a specified path and expects a **200 OK** response. Any other response or timeout marks the VM unhealthy.

---

## Q7. What happens when a health probe fails?

**Answer (explained):**
When a health probe fails:

* The affected VM is marked unhealthy
* New traffic is no longer sent to that VM
* Existing TCP connections may continue (depending on scenario and SKU)
* Traffic is rerouted to healthy VMs automatically

This ensures uninterrupted service.

# üîê Comparing Azure RBAC, Azure Policy & Azure Blueprints

### (Clear, Exam-Ready & Real-World Explanation)

Azure governance is **not handled by a single tool**. Microsoft designed **three complementary services**, each solving a **different layer of control**.

Think of Azure governance like managing a **secure office building**:

* **RBAC** ‚Üí Who can enter which rooms
* **Policy** ‚Üí What rules must rooms follow

---

## 1Ô∏è‚É£ Azure RBAC (Role-Based Access Control)

### üëâ What RBAC REALLY Does

Azure RBAC controls **WHO can do WHAT on WHICH resource**.

It **does NOT** care:

* Whether a resource is secure
* Whether it follows company rules

It ONLY cares about **permissions**.

---

### üß† How RBAC Works

RBAC works using **roles**, assigned to **identities**, at a **scope**.

#### Core Components

* **Roles**

  * Owner ‚Üí Full control
  * Contributor ‚Üí Create & manage resources
  * Reader ‚Üí View only
  * Custom roles ‚Üí Fine-grained control

* **Identities**

  * Users (people)
  * Groups (teams)
  * Service principals (applications)

* **Scopes**

  * Management Group
  * Subscription
  * Resource Group
  * Individual Resource

---

### ‚úÖ RBAC Principles

RBAC follows strict security principles:

* **Least Privilege**
  Give only what‚Äôs necessary, nothing more.

* **Separation of Duties (SoD)**
  Critical actions are split between users.

* **Need-to-Know**
  Access only to required resources.

---

### üìå Real Example

> A developer needs to manage VMs in one resource group.

‚úî Assign **Virtual Machine Contributor**
‚úî Scope ‚Üí Specific resource group
‚ùå No access to storage, networking, or other groups

---

### ‚ùó What RBAC Does NOT Do

* ‚ùå Cannot enforce encryption
* ‚ùå Cannot restrict regions
* ‚ùå Cannot require tagging

üëâ That‚Äôs where **Azure Policy** comes in.

## 2Ô∏è‚É£ Azure Policy

### üëâ What Azure Policy REALLY Does

Azure Policy controls **WHAT is allowed to exist** in Azure.

It **does NOT** care:

* Who created the resource
* Who has access

It ONLY cares about **resource configuration & compliance**.

---

### üß† How Azure Policy Works

Azure Policy evaluates:

* Resource properties
* Deployment settings
* Existing resources

Against:

### üîç Key Capabilities
